IMS PMS Passkey Flow

Passkeys, approvals, key grants, and salary decryption

This document maps the deployed mock and PMS backend from browser click to WebAuthn ceremony, PRF-backed key release, short-lived grant, salary encryption, and audit records. Generated from the current local source on 2026-07-02.

Core Rule

A normal JWT with the ADMIN role is not enough for salary data. PMS salary endpoints require a fresh passkey-derived key grant in X-Key-Grant-Token. Without it they return 403 Key grant is required.

End-to-End Path

The happy path has four phases: enroll a passkey, approve it, authenticate it to mint a grant, then use that grant for salary endpoints.

1

Register

PMS starts a single-use WebAuthn registration ceremony with resident key, user verification, PRF enabled, and phone-first hints.

POST /api/passkeys/registration/options
2

Verify

Browser creates the credential. PMS verifies attestation and stores the public credential metadata. No salary key is released.

POST /api/passkeys/registration/verify
3

Approve

A different admin approves the credential. Future key release checks this approval every time.

POST /api/key-approvals
4

Authenticate

PMS requests a PRF evaluation. The authenticator signs the challenge and returns a PRF output that PMS uses to unwrap the salary DEK.

POST /api/passkeys/authentication/verify
5

Grant

PMS generates a random grant token, stores only its SHA-256 hash, and stores the salary DEK encrypted under a key derived from the grant token.

KeyGrantResponse { token, expiresAt }
6

Salary Read

The mock sends X-Key-Grant-Token. PMS validates ownership, expiry, revocation, and unwraps the DEK into a ThreadLocal for JPA converters.

GET /api/salaries/{employeeExternalId}
7

Salary Write

Create salary uses the same grant gate. Amounts are AES-GCM encrypted before they hit salary_records.net_base_salary.

POST /api/salaries
8

Audit

Successful decrypts, denied decrypts, and salary creates produce append-only audit rows. Amounts are never exposed by the audit endpoint.

GET /api/audit-records?entity=salary&entityId=...

Crypto Tree

The salary value encryption key is a 32-byte DEK. KMS protects the canonical DEK, passkey PRF protects a per-credential wrap, and the short-lived grant token protects a temporary wrap.

1

Canonical salary DEK

If no KMS wrap exists, PMS generates 32 random bytes using SecureRandom. It stores SalaryDekWrap(version=1, recipient=kms, wrappedDek=KMS.encrypt(rawDek)). In local mode, LocalDekWrapper is a test-only prefix wrapper.

2

Passkey PRF wrap

During authentication, PMS derives AES key = SHA-256("ims-passkey-dek-wrap-v1" || 0x00 || passkeyPrfOutput). It stores or reads SalaryDekWrap(version=1, recipient=pk:base64url(SHA-256("ims-passkey-recipient-v1" || 0x00 || credentialId))).

3

Grant token wrap

PMS generates a random 32-byte grant token and derives AES key = SHA-256("ims-key-grant-wrap-v1" || 0x00 || grantTokenUtf8). It stores KeyGrant.wrappedDek = AES-GCM(grantKey, rawDek) and only stores SHA-256(grantToken) for lookup.

4

Salary value encryption

For each salary amount, AesGcmCipher uses AES/GCM/NoPadding, a fresh 12-byte IV, and a 128-bit tag. The database payload is iv || ciphertextAndTag.

Important boundary: WebAuthn private keys and PRF secrets stay on the authenticator. PMS receives the WebAuthn public key during registration and receives the PRF output during authentication. PMS never stores the raw grant token, only its hash and a DEK wrap encrypted under a grant-derived key.

State and Storage

What each table means in the passkey and salary-key model.

passkey_ceremonies

Temporary, single-use WebAuthn request storage.

  • id: ceremony ID sent to the mock.
  • type: REGISTRATION or AUTHENTICATION.
  • request_json: original Yubico request options.
  • Expired after 6 minutes and deleted on verify.

passkey_credentials

Long-lived public credential registration.

  • Credential ID and public key COSE are stored.
  • User identity is external ID plus email.
  • active=false disables the credential.

key_approvals

Dual-control approval by a different admin.

  • One row per credential ID.
  • Inactive or revoked approvals do not release keys.
  • Checked when minting every grant.

salary_dek_wraps

Encrypted copies of the same salary DEK for different recipients.

  • recipient=kms or local is the canonical wrap.
  • recipient=pk:... is a passkey PRF-derived wrap.
  • Unique by (version, recipient).

key_grants

Short-lived server-side grant record.

  • Stores token_hash, never raw token.
  • Stores wrapped_dek encrypted under grant-derived AES key.
  • Bound to user external ID and email.

salary_records and audit_records

Encrypted salary values and append-only audit trail.

  • net_base_salary is BYTEA encrypted via SalaryConverter.
  • Audit old/new values are also encrypted when present.
  • Decrypt audit rows are metadata only.

Failure Map

How common passkey and grant failures surface.

Browser returns NotAllowedError and PMS /verify is never called
The failure happened inside navigator.credentials.create() or navigator.credentials.get(). PMS never received a credential. Usual suspects are user cancellation, phone pairing, Bluetooth/proximity, unavailable provider option, or platform UI failure.
403 Passkey PRF support is required for key release
Registration completed but the credential did not report PRF enabled. PMS refuses to store it for salary key release.
403 Key authority has not approved this passkey
Authentication succeeded, but SalaryKeyValidator.requireKeyApproval() did not find an active approval for that credential/user/email tuple.
403 Key grant is required on salary endpoints
SalaryAccessGrantGuard did not find a valid X-Key-Grant-Token. Missing, expired, revoked, wrong user, wrong email, or unknown token hash all collapse to this error.
403 Salary access requires a passkey-gated salary key
A salary converter tried to encrypt or decrypt without a DEK in SalaryDekHolder. In normal flow this is prevented by @RequiresKeyGrant and the aspect.

Minimal Mental Model

Keep this in your head when debugging the flow.

JWT proves who the admin is.
WebAuthn passkey proves the admin is present and verified.
PRF output lets PMS unwrap the salary DEK for that specific approved credential.
KeyApproval proves another admin allowed that credential to release salary keys.
KeyGrant is a short-lived token that temporarily carries access to the DEK.
SalaryConverter only works while KeyGrantAspect has placed the DEK in SalaryDekHolder.
Without X-Key-Grant-Token, salary endpoints fail before salary data is decrypted.
